# 전남대학교 버스정류장 불편제보 안내 서비스

## PRD (Product Requirements Document)

> 이 문서는 비전공자 기획자가 Antigravity라는 AI 코딩 어시스턴트에게 전달하여 실제 웹서비스 개발을 요청하기 위한 개발 스펙 문서이다. 일반적인 사업/아이디어 기획안이 아니라, 코딩 에이전트가 바로 구현에 착수할 수 있도록 제품 목적, 화면 구조, 데이터 모델, API 연동, 관리자 기능, 비기능 요구사항까지 구체적으로 정의하는 것을 목표로 한다.

## 2. 제품 개요 (Product Overview)

**한 줄 정의**: 전남대학교 주변 버스정류장에서 QR코드를 스캔해 정류장의 불편사항을 즉시 제보하고, 지도에서 정류장 위치와 실시간 도착정보를 확인할 수 있는 웹서비스

**배경 및 문제 정의**: 전남대 주변 버스정류장의 불편사항(벤치 부족, 지붕 없음, 조명 미흡, 정류장 위치 부적절 등)이 있어도 이를 체계적으로 모아 학교/지자체에 전달할 창구가 없음

**목적**: 학생들이 쉽고 빠르게 제보 → 데이터 자동 집계 → 학교/시청/국민신문고에 공식 민원으로 제출

## 3. 타겟 사용자

| 구분 | 대상 | 설명 |
| --- | --- | --- |
| 주 사용자 | 전남대학교 재학생, 통학생 | 모바일에서 QR코드를 스캔하여 특정 정류장에 대한 불편사항을 빠르게 제보하는 사용자 |
| 부 사용자 | 서비스 운영자 | 제보 데이터를 취합하고 민원을 제출하는 관리자. 예: 총학생회 또는 기획자 본인 |

---

## 4. 서비스 범위 (In Scope / Out of Scope)

### In Scope

- QR코드 기반 정류장별 제보 폼
- 카카오맵 API 기반 정류장 지도 (마커 표시, 클릭시 정보창)
- 마커 클릭 시 실시간 버스 도착예정정보 텍스트 표시 (TAGO API, 클릭 시점에 1회 조회)
- 제보 데이터 자동 집계 및 관리자용 대시보드/리포트
- 민원 제출용 문서 템플릿 자동 생성

### Out of Scope

- 지도 위에서 버스가 실시간으로 움직이는 애니메이션 기능 제외
- 로그인/회원가입 기능 제외
- 모바일 앱(네이티브) 개발 제외, 반응형 웹만 개발

---

## 5. 핵심 사용자 시나리오 (User Flow)

**시나리오 1 - QR 제보**
1. 학생이 정류장 QR코드를 스캔
2. stop_id 기반 제보 폼 진입
3. 불편 유형 선택
4. 설명/사진 입력
5. 제출 후 완료 화면 표시

**시나리오 2 - 지도로 확인**
1. 메인 지도 진입
2. 정류장 마커 확인
3. 마커 클릭 시 정류장명/누적 제보수/버튼 표시
4. 도착정보 보기 클릭 시 텍스트 조회
5. 제보하기 클릭 시 폼 이동

**시나리오 3 - 관리자 데이터 확인**
1. 관리자 접속
2. 정류장별/유형별 제보 집계 확인
3. 기간 필터 리포트 생성
4. 민원 문서 템플릿 다운로드

## 6. 화면 구성 (Page/Screen List)

| 화면명 | 경로(예시) | 설명 |
| --- | --- | --- |
| 메인/지도 화면 | `/` | 카카오맵 기반 정류장 지도, 마커, 정보창 |
| 제보 폼 화면 | `/report?stop_id=xxx` | QR 스캔 또는 지도에서 진입, 불편 유형 선택 및 제출 |
| 제출 완료 화면 | `/report/complete` | 제보 확인 메시지 |
| 관리자 대시보드 | `/admin` | 로그인 필요(간단한 비밀번호 방식), 집계 데이터/차트 |
| 관리자 리포트 생성 | `/admin/report` | 기간별 필터, PDF 다운로드 |

## 8. 외부 API 연동 명세

- **카카오맵 API** (Kakao Maps JavaScript SDK): 지도 렌더링, 마커, 인포윈도우. JavaScript 키 필요
- **국가대중교통정보센터 API** (공공데이터포털, TAGO): 정류소별 도착예정정보 조회. 서비스키 필요

> API 키는 코드에 하드코딩하지 말고 환경변수(`.env`) 처리할 것

---

## 9. 비기능 요구사항

- 반응형 웹(모바일 우선, QR스캔은 대부분 모바일에서 이루어짐)
- 익명 제보이나 스팸/도배 방지를 위한 최소한의 제한 고려
- 무료/저비용 운영 가능한 스택 우선 고려
- 개인정보 최소 수집 원칙 준수

---

## 10. 기술 스택 제안

- **프론트엔드**: React 기반 (Next.js 권장) 또는 Antigravity가 판단하는 적합한 프레임워크
- **백엔드/DB**: 서버리스 함수 + 경량 DB(예: Supabase, Firebase 등) - Antigravity 판단에 위임하되 이유 명시 요청

> 이 항목은 "Antigravity가 프로젝트 상황에 맞게 자유롭게 제안 가능"

## 11. 마일스톤 (개발 단계)

| 단계 | 내용 |
| --- | --- |
| 1단계 | 지도 화면 + 정류장 마커 표시 (정적 데이터) |

## 12. Antigravity에게 전달할 작업 지시 예시 문구

1. "이 PRD를 참고해서 1단계부터 순서대로 구현해줘. 먼저 지도 화면부터 만들어줘."
2. "이 PRD 기준으로 Next.js 기반 프로젝트 구조를 먼저 제안해줘. 폴더 구조, 환경변수 목록, 데이터 모델, API 라우트 설계안을 설명한 뒤 구현을 시작해줘."
3. "QR 제보 폼 기능부터 구현해줘. `/report?stop_id=xxx` 형태로 정류장을 자동 식별하고, 제보 저장과 완료 화면까지 동작하도록 만들어줘."
4. "관리자 대시보드 MVP를 구현해줘. 정류장별/유형별 제보 건수 표와 기간 필터, CSV 또는 PDF 다운로드 초안을 포함해줘."
